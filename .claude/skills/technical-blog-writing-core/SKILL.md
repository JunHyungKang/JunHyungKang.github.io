# Technical Blog Writing Core

개발자 독자를 위한 기술 블로그 작성·리라이트 절차를 제공한다.
특정 외부 CLI나 SaaS 명령에 의존하지 않고, 구조·근거·재현성을 우선한다.

## 사용 시점

다음 요청에서 사용한다.

- 기술 블로그 초안 작성
- 기존 기술 글 리라이트/교정
- 튜토리얼 구조 개선
- 딥다이브/비교/회고 글 구성

## 목표

다음 기준을 만족하도록 작성한다.

- 문제와 해결을 빠르게 이해 가능
- 코드 예제를 실제로 재현 가능
- 주장에 수치/근거 포함
- 트레이드오프와 한계 명시

## 글 유형 선택

요청 의도에 맞는 유형을 먼저 선택한다.

1. Tutorial / How-to: 단계별 구현 안내
2. Deep Dive / Explainer: 개념·원리·내부 동작 설명
3. Postmortem / Incident: 장애 원인·대응·재발 방지
4. Benchmark / Comparison: 방법론 기반 비교와 해석
5. Architecture / System Design: 선택 근거와 트레이드오프

## 기본 작성 절차

1. 독자와 전제 정의
   - 대상 독자(입문/중급/고급) 명시
   - 사전 지식, 런타임/프레임워크 버전 명시
2. 문제 정의
   - 핵심 문제를 1~2문장으로 제시
   - 왜 중요한지 수치/실패 사례로 연결
3. 아웃라인 구성
   - 문제 → 접근 → 구현/검증 → 결과 → 한계 → 결론
4. 본문 작성
   - 섹션마다 주장 → 근거 → 예시 순서 유지
5. 최종 검증
   - 체크리스트로 구조·근거·재현성 점검

## 권장 본문 구조

```markdown
# 제목 (결과/효과가 드러나게)

**TL;DR:** 핵심 요약 2~3문장

## 문제 정의
- 현재 상태, 병목, 실패 맥락

## 접근 방법
- 선택한 방법과 선택 이유

## 구현
- 단계별 설명 + 코드 + 실행 결과

## 결과
- 측정 수치, 비교 표/그래프, 관찰

## 트레이드오프와 한계
- 적용이 불리한 조건, 운영 비용

## 결론
- 적용 기준, 다음 액션

## 참고 자료
- 공식 문서, 벤치마크, 관련 글
```

## 개발자 글쓰기 규칙

### 톤

- 단정적으로 쓰되 과장하지 않기
- 마케팅 표현보다 기술 근거 우선
- 대안과 한계를 숨기지 않기

### 권장 표현

- "p99 지연시간 820ms→95ms 감소"
- "복잡도는 증가하지만 콜드스타트 비용은 감소"
- "Node 20.11 / React 19 기준 검증"

### 금지 표현

- "요즘 빠르게 변화하는 시대에"
- "그냥/쉽게/simply"
- "당연히/obviously"
- 근거 없는 "혁신적", "압도적"

## 코드 예시 규칙

모든 코드 블록은 재현성을 기준으로 작성한다.

- 언어 식별자 포함 (```ts, ```py)
- 실행 전제(버전/의존성) 명시
- 복붙 실행 가능한 최소 완결 예시 우선
- 예상 출력/결과 함께 제시
- 실패 케이스 또는 에러 처리 최소 1개 포함

## 결과 섹션 규칙

가능하면 정량 수치를 포함한다.

- Before/After
- 측정 조건(데이터 크기, 횟수, 런타임)
- 평균뿐 아니라 p95/p99 같은 tail 지표

수치가 없다면 관찰 근거를 명시한다.

- 로그 캡처
- 재현 절차
- 사용자 영향 범위

## 트레이드오프 필수 규칙

모든 기술 선택의 비용을 명시한다.

- 도입 복잡도
- 운영/디버깅 난이도
- 런타임/벤더 종속성
- 성능과 생산성의 균형

## 리라이트 절차

1. 구조 교정: 결론/핵심을 앞당기기
2. 문장 교정: 한 문장 한 주장 원칙 유지
3. 근거 교정: 근거 없는 형용사 제거
4. 재현성 교정: 버전/입출력/전제 누락 보완
5. 독자성 교정: 대상 독자 수준에 맞게 깊이 조정

## 출력 형식

요청이 "초안 작성"이면 다음 순서로 출력한다.

1. 제목 후보 3개
2. TL;DR
3. 섹션별 아웃라인
4. 본문 초안
5. 체크리스트 기준 자체 점검 메모

요청이 "리라이트"이면 다음 순서로 출력한다.

1. 문제점 요약(구조/근거/표현)
2. 개선된 본문
3. 변경 이유 요약

## 최종 체크리스트

- [ ] 제목에서 결과가 드러나는가?
- [ ] TL;DR이 2~3문장으로 명확한가?
- [ ] 코드 예제가 실제 실행 가능한가?
- [ ] 버전/전제가 명시되었는가?
- [ ] 측정 근거(수치/방법)가 있는가?
- [ ] 트레이드오프/한계를 숨기지 않았는가?
- [ ] 과장 표현을 제거했는가?
- [ ] 다음 액션(적용 기준/확장 포인트)이 있는가?

## 제약

- 특정 외부 CLI 설치/실행 절차를 전제로 하지 않는다.
- 특정 서비스 벤더 종속 명령 대신 일반 원칙을 사용한다.
- 기술 정확성과 독자 신뢰를 최우선으로 유지한다.
